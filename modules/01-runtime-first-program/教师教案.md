# M1 · 运行环境与第一个程序 · 教师教案

> 状态：初稿，待教师审核  
> 建议时长：25 分钟  
> 最低时长：20 分钟  
> 扩展上限：40 分钟  
> 前置要求：M0 · 课程介绍与编程概述  
> 后续衔接：M2 · 值、变量、表达式与基础 Debug

## 1. 模块定位

M1 承接 M0 结尾建立的“修改—运行—观察”反馈闭环，让学生第一次亲手创建、保存、运行并修改一个真正属于自己的 Python 程序。本节不做系统语法教学，也不把安装步骤讲成一堂课。核心任务是帮学生建立三条最基本的操作直觉：

1. 代码写在一个 `.py` 文件里，是可以反复编辑和保存的文本。
2. 运行是一个明确的动作，由 Python 解释器执行当前文件，产生输出。
3. 修改代码并重新运行，输出会随之改变；确认“我正在编辑的文件”就是“实际被运行的文件”。

本节的第一个程序不是随意的 `Hello World`，而是贯穿式生活项目“我的一周预算助手”的欢迎程序。学生从第一行代码起就进入六天主线，后续每天都在同一个问题情境里继续推进。

M1 只处理“让第一个程序成功运行”所必需的问题。安装救援、备用环境切换属于本节职责；Traceback、错误类型和系统化排错留给 M2 后半。

## 2. 学习目标

### 2.1 知识目标

学生能够：

1. 说明 Python 解释器、`.py` 文件、编辑器和运行结果之间的关系。
2. 按课前发放的安装指南确认本机 Python 与编辑器可用，或在失败时切换到备用环境。
3. 独立创建、保存、运行并重新运行一个程序。
4. 认识 `print()`、字符串和“从上到下逐行执行”的最小直观模型。

### 2.2 思维目标

学生能够：

1. 区分代码文件、运行命令和输出结果三者的不同角色。
2. 通过“修改后重新运行”，建立“代码变化导致程序行为变化”的最初反馈闭环。
3. 确认当前正在编辑的文件与实际运行的文件一致，避免“改了没生效”的常见困惑。

## 3. 达成证据

本节至少收集以下课堂证据：

- 学生能在自己的环境里创建并保存一个 `.py` 文件，并成功运行看到预期输出。
- 学生能修改字符串内容后重新运行，并观察到输出随之改变。
- 学生能说出“为什么这次输出没变”通常是因为没保存、或运行的不是当前文件。
- 学生能用一句话说明解释器、文件和输出各自是什么。

不以学生记住安装器选项、菜单位置或操作系统配置细节作为达成标准。

## 4. 教学边界

### 本节必须完成

- 一次环境确认：解释器可用、编辑文件、运行结果三者对得上。
- 现场安装救援或备用环境切换（针对课前未成功的学生）。
- 独立创建、保存、运行一个最小程序。
- 修改字符串后重新运行，观察输出变化。
- 一次“确认编辑文件 = 运行文件 = 实际输出”的核对。

### 本节不展开

- Traceback 的结构、错误类型和系统化排错流程（留给 M2 后半）。
- 变量、类型、`input()`、表达式和 f-string（M2）。
- `print()`、字符串、括号和引号的正式语法规则（M2 系统讲）。
- 虚拟环境、包管理、`pip`、命令行系统知识。
- IDE 全功能巡览、插件、调试器、Git 集成等。

“不展开”表示暂时移出本节主线，不表示这些内容不重要。

## 5. 课前准备

- 课前 3–5 天发放《环境安装指南》，要求学生先自行安装 Python 与编辑器，并尝试运行一行 `print()`。
- 收集课前安装反馈，标记出未成功或不确定的学生，课上优先支援。
- 教师机准备好可投影的编辑器与终端，字号放大到后排可见。
- 准备**备用环境**方案（任选其一并提前验证）：
  - 预装好的教室机器或 U 盘便携环境；
  - 在线运行环境（如浏览器内 Python），用于设备暂时无法安装的学生。
- 准备 M0 已出现过的两行 `print()` 代码，作为第一个程序的起点。
- 预留板书区域记录三个词：**解释器**、**文件**、**输出**，以及一列“改了没生效”的常见原因。

## 7. 分环节教学稿

### 7.1 从 M0 接线

上一节最后，我们看到“改一行代码、重新运行、输出就变了”。那一次是老师在演示。这一节，你要亲手把它做一遍——创建一个属于你的文件，运行它，再改它。

先快速确认一下：课前按指南装好 Python 和编辑器、并成功运行过一行代码的同学，请举手。

> 没装好也完全没关系。设备问题不应该挡住你学编程。我们准备了备用环境，先让你能运行起来，安装可以课后再补。

对未成功的学生：先登记，随后由助教引导切换到备用环境，不在全班面前逐台排查安装。

### 7.2 解释器、文件、编辑器、输出

先建立四个角色，不展开细节：

- **`.py` 文件**：你写的代码，本质是一段可以反复编辑和保存的文本。
- **编辑器**：你用来写和改这段文本的工具。
- **Python 解释器**：真正“读懂并执行”这段代码的程序。
- **运行结果（输出）**：解释器执行后在屏幕上产生的东西。

用一句话串起来：

> 你在**编辑器**里写好 `.py` **文件**，让 **Python 解释器**去运行它，屏幕上出现的就是**输出**。

现场演示一次完整动作：打开一个已有文件 → 运行 → 指着屏幕说“这是文件、这是运行按钮或命令、这是输出”。演示时不解释代码语法，只强调“运行是一个明确的动作”。

板书三个词，本节反复回指：

```text
解释器   文件   输出
```

### 7.3 独立创建第一个程序

给出明确任务，让学生自己动手：

> 创建一个新文件，命名为 `budget_hello.py`，输入下面这行代码，保存后运行它。

```python
print("欢迎使用我的一周预算助手")
```

运行后应看到：

```text
欢迎使用我的一周预算助手
```

巡视要点：

- 文件是否真的保存了（很多编辑器未保存时标题栏有提示）。
- 引号是否为英文半角引号，而不是中文全角引号。
- 运行方式是否正确（编辑器运行按钮或终端 `python budget_hello.py`）。

对卡住的学生，先让程序跑起来，不纠结“最规范”的做法。这一步的成功标准只有一个：**屏幕上出现了那句欢迎语**。

说明这行代码的最小模型即可，不展开语法：

> `print(...)` 的作用是把括号里引号中的文字显示到屏幕上。程序从上到下、一行一行执行。

### 7.4 修改—运行—观察

现在亲手完成 M0 演示过的闭环。要求学生把欢迎语改成自己的版本，例如：

```python
print("欢迎使用我的一周预算助手，先看看这周还能花多少")
```

保存、重新运行，观察输出是否随之改变。

提炼闭环：

```text
修改代码  ->  保存  ->  重新运行  ->  观察输出变化
```

强调一句：

> 代码不改、不重新运行，输出就不会变。程序不会“猜”你心里想的内容，它只执行你保存并运行的那一版。

### 7.5 确认运行的是哪个文件

这是初学者最常见、也最容易自我怀疑的问题：“我明明改了，为什么输出没变？”故意制造一次这种情况，让学生学会自己诊断。

演示两种典型原因：

1. **改了但没保存**：解释器运行的仍是上次保存的版本。
2. **运行了另一个文件**：编辑的是 A 文件，实际运行的是 B 文件。

引导学生形成一个简单的自查清单：

```text
输出没变，先问自己：
1. 我保存了吗？
2. 我运行的，是不是我正在改的这个文件？
```

让学生刻意复现一次“不保存就运行”，看到输出没变，再保存重跑，看到输出变化。这样他们**主动**经历过一次，比只听讲更牢固。

### 7.6 收束与衔接

用三个词收束：

> 今天你确认了三样东西——**解释器**在运行、**文件**是你能改的代码、**输出**是运行的结果。你已经能独立创建、运行和修改一个程序了。

预告 M2：

> 现在这个程序每次只会说同一句话。下一节，我们让它开始“记住”一周预算是多少、已经花了多少——这就需要**变量**。

## 8. 本章小结

请学生独立回答，再与同伴互查：

1. 一句话说明：解释器、`.py` 文件和输出结果，各自是什么？
2. 你改了代码、重新运行，输出却没变。可能的两个原因是什么？
3. `print("...")` 里引号中的内容，在运行时会发生什么？

### 参考答案

1. `.py` 文件是可编辑保存的代码文本；解释器是执行这段代码的程序；输出是执行后显示在屏幕上的结果。
2. 一是没保存，运行的还是旧版本；二是运行的不是当前正在编辑的文件。
3. 引号中的文字会被原样显示到屏幕上（不含引号本身）。

如果多数学生仍分不清“文件”和“运行动作”，M2 开头再用同一个 `budget_hello.py` 做一次“改—存—跑”对照。如果多数学生栽在中文全角引号，M2 讲字符串时优先强调引号种类。

## 9. 常见误区与处理

| 误区 | 处理方式 |
| --- | --- |
| “代码写好就自动生效了。” | 强调运行是一个明确动作；不运行，代码只是躺着的文本。 |
| “我改了，为什么没变？” | 带学生走自查清单：先看保存，再看运行的是不是当前文件。 |
| 用中文全角引号 `“ ”` | 指出必须用英文半角引号 `" "`；这是第一个高频卡点。 |
| “必须先把环境装到完美才能学。” | 先用备用环境跑起来，安装问题课后补，不阻断学习。 |
| “输出里怎么没有引号？” | 说明引号是代码里的边界标记，不是要显示的内容。 |
| 把编辑器等同于 Python | 区分“写代码的工具”和“执行代码的解释器”。 |

## 10. 时间调整

### 10.1 压缩到 20 分钟

- 三者关系用一张图讲完，减少现场演示的解说。
- 修改—运行—观察只做一轮。
- “确认运行文件”合并进巡视，由助教对个别学生处理，不全班演示。
- 必须保留：独立创建运行、修改后重跑看到变化、一次运行文件确认。

### 10.2 扩展到 40 分钟

- 让学生再加一行 `print()`，预测两行的输出顺序，验证“从上到下执行”。
- 让学生故意把英文引号改成中文引号，观察会发生什么（为 M2 的报错做铺垫，但本节不展开 Traceback）。
- 让学生把欢迎语改成两三种不同版本，体会“同一个程序、不同输出”。
- 简单演示终端与编辑器两种运行方式的对应关系。

扩展内容只在核心小结题已经能通过时加入。故意制造的错误必须明确说明是“故意的”，并在本节内恢复到可运行状态。

## 11. 助教任务

- 开课前汇总课前安装反馈，标出需要优先支援的学生。
- 环节 7.1 后，立即引导未成功的学生切换备用环境，让其先能运行程序。
- 巡视 7.3，确保**每位**学生至少成功运行一次；重点排查未保存和全角引号。
- 7.5 中观察学生能否自己走自查清单，而不是直接替他们修好。
- 把两类共性问题反馈给主讲：环境类（装不上、跑不起来）和操作类（不保存、跑错文件）。

## 12. 与 M2 的接口

M1 结束时，学生已经能独立创建、保存、运行和修改程序，并区分文件、运行动作和输出结果，但程序还只能打印固定文字。M2 从这里继续：

- 让 `budget_hello.py` 从“只会说一句话”变成“能记住一周预算、总支出等状态”，自然引出**变量、赋值和类型**。
- 在学生已经接触值、变量、类型和表达式之后，M2 后半再系统讲 Traceback、错误类型和基础 Debug。

后续模块继续反复使用这条主线：

```text
明确问题
-> 明确输入、输出、规则和约束
-> 构造具体例子
-> 拆分可执行步骤
-> 编写最小可运行版本
-> 追踪、测试、修改并解释
```

## 13. 本轮审核重点

请优先判断：

1. 25 分钟内“确保每位学生独立运行成功”是否现实，救援与备用环境安排是否足够。
2. 把第一个程序定为预算助手欢迎程序、直接进入主线，是否合适。
3. “确认运行文件”这一环节是否值得单列，还是可以并入巡视。
4. 环境确认的深度是否恰当——够用但不越界到 M2。
5. M1 与 M0、M2 的边界是否清楚，尤其是“不讲 Traceback”的分界。
