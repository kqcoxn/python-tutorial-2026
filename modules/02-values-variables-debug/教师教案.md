## 1. 模块定位

M2 承接 M1 结尾的固定字符串程序，让学生第一次让程序“记住”状态，并开始写出可计算、可输入、可格式化输出的预算面板。本节严格分为前后两半：

1. **前半（约 45 分钟）**：值与四种基本类型 → 变量与赋值/重新赋值/命名 → 表达式与运算 → 字符串/引号/转义/拼接/f-string → 注释 → `input()`/`type()`/`len()`/类型转换 → 独立完成“输入—计算—格式化输出”程序。
2. **后半（约 35 分钟）**：把报错当证据 → Traceback 结构（行号、最后一行）→ 四种错误类型（SyntaxError/NameError/TypeError/ValueError）结合已学语法 → “观察—提出一个假设—只改一处—重新验证”排错流程 → 区分“无法运行”与“运行但结果错” → 独立完成一次有依据的排错复测。

贯穿项目仍是“我的一周预算助手”：从 M1 的 `budget_hello.py` 演进到 `budget_d2_basics.py`，再进入 `budget_d3_record.py` 雏形。按六天路线，M2 分布在第 1 天末（30 分钟）+ 第 2 天（50 分钟）；实施计划将其映射为“第 2 课建立预算状态面板、第 3 课记录第一笔支出并做基础 Debug”。

M2 只处理值、变量、表达式、基础输入输出与系统化基础 Debug。条件判断留给 M3；浮点误差、位运算、eval、高级格式协议等一律不展开。

## 2. 学习目标

### 2.1 知识目标

学生能够：

1. 识别并说明四种基本类型：`int`、`float`、`str`、`bool`；用 `type()` 观察字面量与变量的类型。
2. 使用变量完成赋值、重新赋值与合理命名（snake_case）；理解变量保存的是当前值。
3. 读写常见算术表达式与复合赋值；预测表达式的结果与类型；在需要时使用括号明确运算顺序。
4. 正确使用字符串引号、转义、拼接与 f-string 格式（含 `:.0%`、`:.2f`）；为关键决策写注释。
5. 说明 `input()` 永远返回字符串；用 `int()`/`float()`/`str()` 做必要转换；用 `len()` 观察字符串长度。
6. 阅读 Traceback 的行号与最后一行；识别 SyntaxError、NameError、TypeError、ValueError 与已学语法的对应关系。

### 2.2 思维目标

学生能够：

1. 区分“值本身”“类型”和“屏幕上的显示形式”。
2. 用手动追踪表跟进多次赋值后的变量状态。
3. 把报错当作证据而非惩罚：从下往上读 Traceback，提出一个假设，只改一处后复测。
4. 区分“无法运行”（如 SyntaxError）与“运行但结果错 / 输入不合法”（如逻辑错、ValueError）。

## 7. 分环节教学稿

### 7.1 从 M1 接线

上一节，学生已经能创建、运行并修改：

```python
print("欢迎使用我的一周预算助手")
```

指出局限：

> 这个程序每次只说同一句话。它不知道“本周预算是多少”“已经花了多少”。要让程序**记住**状态，就需要**变量**。

板书本节主线：值 → 变量 → 表达式 → 输入转换 → 报错当证据。

### 7.2 值与四种基本类型

先建立“一个值有它的类型”：

| 类型 | 含义（学生语言） | 字面量例子 |
| --- | --- | --- |
| `int` | 整数 | `500`、`180` |
| `float` | 小数 | `3.5`、`10 / 3` 的结果 |
| `str` | 文本 | `"30"`、`"欢迎"` |
| `bool` | 真/假 | `True`、`False` |

现场运行观察：

```python
print(type(30), type(3.5), type("30"), type(True))
```

实测：

```text
<class 'int'> <class 'float'> <class 'str'> <class 'bool'>
```

强调：`"30"` 和 `30` 看起来都像数字，但类型不同——后面混用就会报错。

### 7.3 变量、赋值、重新赋值、命名

让程序开始“记住”预算状态：

```python
weekly_budget = 500
total_spent = 180
remaining = weekly_budget - total_spent
print("本周预算：", weekly_budget)
print("已花：", total_spent)
print("剩余：", remaining)
```

实测输出：

```text
本周预算： 500
已花： 180
剩余： 320
```

命名约定：使用 snake_case（如 `total_spent`），名称要能读懂；本节不讨论类名风格。

**重新赋值 + 复合赋值（手动追踪素材）**：

```python
total_spent = 180
total_spent = total_spent + 45   # 又花了 45
total_spent += 30                # 再花 30
print(total_spent)               # 255
```

请学生填写追踪表（先写预测，再运行验证）：

| 步骤 | 语句 | total_spent |
| --- | --- | ---: |
| 1 | total_spent = 180 | 180 |
| 2 | total_spent = total_spent + 45 | 225 |
| 3 | total_spent += 30 | 255 |

讲解点：

- `=` 是赋值，不是数学里的“恒等”。
- 重新赋值会覆盖旧值；复合赋值 `+=` 是“在旧值上改”。
- `==` 比较留给 M3，本节只用 `=`。

### 7.4 表达式与运算

用预算相关数字做预测，再运行：

```python
print(7 // 2, 7 % 2, 2 ** 10)   # 3 1 1024
print(10 / 3)                    # 3.3333333333333335
print("a" + "b", "ab" * 3)       # ab ababab
print(type(30), type(3.5), type("30"), type(True))
```

实测：`3 1 1024` / `3.3333333333333335` / `ab ababab` / `<class 'int'> <class 'float'> <class 'str'> <class 'bool'>`

简要约定：

- `//` 整除，`%` 取余，`**` 乘方，`/` 除法（结果常为 float）。
- 运算有顺序；拿不准时加括号。
- **浮点现象一句话（不展开）**：`10 / 3` 结果很长，是因为小数在计算机里常常只能近似保存；需要好看时用格式化，例如 `:.2f`。

### 7.5 字符串与 f-string

回顾 M1 的引号要求：必须用英文半角引号。补充：

- 拼接：`"a" + "b"` → `ab`
- 重复：`"ab" * 3` → `ababab`
- 转义：需要在字符串里写引号等特殊字符时使用（点到为止，不展开全部转义表）。
- 注释：用 `#` 写目的、假设或关键决策，而不是复述代码字面意思。

**前半独立编写目标：`budget_d2_basics.py`**

```python
weekly_budget = 500
total_spent = 180
remaining = weekly_budget - total_spent
spending_ratio = total_spent / weekly_budget

print(f"本周预算：{weekly_budget} 元")
print(f"累计支出：{total_spent} 元")
print(f"剩余预算：{remaining} 元")
print(f"已使用：{spending_ratio:.0%}")
```

实测输出：

```text
本周预算：500 元
累计支出：180 元
剩余预算：320 元
已使用：36%
```

点明：`:.0%` 把比例显示为百分比；需要两位小数金额时可用 `:.2f`（扩展时可多练）。

### 7.6 输入与类型转换

内置函数与转换（先小块演示）：

```python
print(int("30"), float("3.5"), str(30))   # 30 3.5 30
print(len("abc"))                          # 3
print(type(30))                            # <class 'int'>
```

**加入输入（`budget_d3_record.py` 雏形）**：

```python
weekly_budget = int(input("请输入本周预算："))
amount = int(input("请输入一笔支出金额："))
total_spent = amount
remaining = weekly_budget - total_spent
spending_ratio = total_spent / weekly_budget

print(f"累计支出：{total_spent} 元")
print(f"剩余预算：{remaining} 元")
print(f"已使用：{spending_ratio:.0%}")
```

关键讲解点：

> `input()` 永远返回字符串，必须用 `int()` 转换后才能参与算术。

巡视时重点看：是否写成 `amount = input(...)` 后直接相减。若出现，先记现象，后半用真实 Traceback 处理。

### 7.7 前半小结 + 转入 Debug

收束前半三点：

1. **值**：程序处理的数据本身（500、`"30"`、True…）。
2. **类型**：值属于哪一类（int/float/str/bool）。
3. **显示形式**：同一数值可用不同格式打印（如比例 → `36%`）。

抛出混用现象，自然转入后半：

> 如果忘记把输入转成数字，程序会怎样？我们不靠猜——读报错。

### 7.8 读懂 Traceback

**Traceback 阅读要点**：

1. 从下往上读，最后一行是错误类型 + 简述。
2. 倒数第二区块的 `line N` 指出出错行号（本机文件里可直接跳转）。
3. 错误类型对应本节已学语法：
   - **SyntaxError**：写法本身不合法，运行前就报（引号/括号/冒号）。
   - **NameError**：用了没定义或拼错的名字。
   - **TypeError**：类型不匹配，如 `int - str`。
   - **ValueError**：类型对，但值不合法，如 `int("30元")`。
4. 排错流程：观察证据 → 提出一个假设 → 只改一处 → 重新运行验证。一次只改一处，才能确认到底是哪个改动生效。

#### 故障示例 1：TypeError（字符串与数字混用 —— 忘记转换）

**故意演示 / 故意错误**（本节内修复后恢复可运行）：

```python
weekly_budget = 500
amount = input("请输入一笔支出金额：")   # 故意漏掉 int()，此时 amount 是字符串
remaining = weekly_budget - amount        # 出错行
print(remaining)
```

真实 Traceback（实测）：

```text
Traceback (most recent call last):
  File "budget_d3_record.py", line 3, in <module>
    remaining = weekly_budget - amount
                ~~~~~~~~~~~~~~~^~~~~~~~
TypeError: unsupported operand type(s) for -: 'int' and 'str'
```

修复：把输入包进 `int(...)`，**只改一处** → `amount = int(input("请输入一笔支出金额："))`。

#### 故障示例 2：ValueError（转换了，但内容不是纯数字）

**故意演示 / 故意错误**：

```python
amount = int("30元")   # 用户多打了单位
```

真实 Traceback（实测，末行）：

```text
ValueError: invalid literal for int() with base 10: '30元'
```

讲解：这是“运行但输入不合法”，与 TypeError 不同。完整的输入校验留给后续（M3/扩展 try-except），本节只要求读懂并定位。

#### 故障示例 3：NameError（变量名拼写错误）

**故意演示 / 故意错误**：

```python
total_spent = 180
print(totl_spent)   # 拼写错误
```

真实 Traceback（实测，末行）：

```text
NameError: name 'totl_spent' is not defined
```

#### 故障示例 4：SyntaxError（括号/引号不配对 —— 无法运行）

**故意演示 / 故意错误**：

```python
print("剩余预算：, remaining)   # 少了一个引号
```

说明：SyntaxError 在运行前就报出，程序根本没开始执行，用来对照“无法运行 vs 运行但结果错”。

每个故意错误演示后，立即恢复为可运行版本，避免学生文件卡在坏状态。

### 7.9 排错流程

板书并带领走一遍：

```text
观察证据（最后一行 + 行号）
  -> 提出一个假设（只猜一处原因）
  -> 只改一处
  -> 重新运行验证
```

学生任务：在 `budget_d3_record.py` 相关代码中，定位 TypeError（或教师指定的一处故意错误），只改一处并复测成功。

强调：

- 看到红字先读最后一行，不要慌。
- 一次改好几处再运行，就不知道哪处起作用。
- 不用 `eval` 转换输入；用 `int()`/`float()`。

### 7.10 收束与衔接

三点总结：

1. 变量让程序记住状态；类型决定能做什么运算。
2. `input()` 是字符串，参与算术前必须转换。
3. 报错是证据：读行号与最后一行，一次只改一处再验证。

预告 M3：

> 现在程序能算出剩余预算，但还不会回答“超支了吗？”“要不要提醒？”——下一节用**条件判断**让程序按情况做不同的事。

## 8. 本章小结

请学生独立回答，再与同伴互查：

1. `input()` 返回什么类型？要参与加减需要先做什么？
2. 追踪：`x=10; x+=5; x=x*2` 后 x 是多少？
3. 看到 `TypeError: unsupported operand type(s) for -: 'int' and 'str'`，最可能哪里出了问题？
4. 排错四步是什么？为什么一次只改一处？

### 参考答案

1. `input()` 永远返回 `str`（字符串）。要参与加减，需先用 `int()` 或 `float()` 转换成数字。
2. `x` 最终是 `30`（10 → 15 → 30）。
3. 最可能是某处把字符串和整数做了减法——常见原因是 `input()` 后忘记 `int()`/`float()` 转换。
4. 观察证据 → 提出一个假设 → 只改一处 → 重新运行验证。一次只改一处，才能确认到底是哪个改动生效。

如果多数学生仍认为 `input()` 直接得到数字，M3 开头用 `type(input(...))` 再演示一次。如果多数学生一次改多处，助教在巡视中强制“先说出假设，再改一行”。

## 9. 常见误区与处理

| 误区 | 处理方式 |
| --- | --- |
| input() 拿到的是数字 | 演示 type()，说明它永远是 str，要 int()/float() |
| 中文全角引号/括号 | 对照 SyntaxError；延续 M1 的引号强调 |
| 一次改好几处再运行 | 强调只改一处再验证，否则不知道哪处起作用 |
| 看到红字就慌 | 报错是证据不是惩罚；先读最后一行 |
| = 与 == 混淆 | 本节只用 =（赋值）；== 留到 M3 |
| 用 eval 转换输入 | 明确不用 eval，用 int()/float() |

## 10. 时间调整

### 10.1 压缩到 55 分钟

- 压缩表达式广度、转义细节、扩展格式（`:.2f` 可只提一句）。
- 必须保留：四类型、变量与追踪、输入输出、必要转换、核心运算、f-string、一次读报错 + 一次最小排错。
- 预测题减少轮次；ValueError/NameError/SyntaxError 以对照表快速过，深度演示聚焦 TypeError。
- 独立编写与独立排错各保留一轮，不取消。

### 10.2 扩展到 100 分钟

- 加浮点现象一例（仍不系统讲误差理论），练习 `:.2f`。
- 复合赋值更多形态（`-=`、`*=` 等）与运算顺序括号题。
- 断点单步演示（扩展档末尾，非必须达成）。
- 更多故障类型练习：学生轮流读 Traceback 并只改一处。
- 扩展内容只在核心达成证据已通过时加入。故意制造的错误必须标注“故意演示/故意错误”，并在本节内恢复可运行。

## 11. 助教任务

助教为梁健（第 1–3 天）。

- 开课前确认学生 M1 环境仍可用；协助打开本节工作目录。
- 前半巡视：追踪表填写、snake_case 命名、半角引号、f-string 是否跑通。
- 输入环节重点盯：是否对 `input()` 做了 `int()`/`float()`；禁止引导学生使用 `eval`。
- 后半巡视：确保学生**先读最后一行与行号**，再提出一个假设；坚持“只改一处再运行”。
- 故意错误演示后，协助把学生文件恢复为可运行状态。
- 把共性问题反馈给主讲：未转换输入、全角符号、一次改多处、看到红字直接重写整段。

## 12. 与 M3 的接口

M2 结束时，学生已经能用变量保存预算状态，完成“输入—计算—格式化输出”，并能依据 Traceback 做最小有依据修复。程序仍不会按条件分支：

- M3 从“剩余多少 / 是否超支 / 是否需要提醒”等问题自然引出**条件判断**。
- 本节明确未讲的 `==`、比较运算与分支结构，在 M3 系统展开。
- 输入不合法时的完整校验与更系统的异常处理，可在 M3 或后续扩展中衔接。

后续模块继续反复使用这条主线：

```text
明确问题
-> 明确输入、输出、规则和约束
-> 构造具体例子
-> 拆分可执行步骤
-> 编写最小可运行版本
-> 追踪、测试、修改并解释
```

## 13. 本轮审核重点

请优先判断：

1. 80 分钟两段式（前半语法约 45 + 后半 Debug 约 35）是否可执行；55/100 压缩扩展是否可操作。
2. 代码演进与真实 Traceback 是否与骨架一致、是否全部可复现。
3. 故障示例是否均标注“故意演示/故意错误”，且本节内恢复可运行的要求是否写清。
4. 教学边界是否清楚：不用 eval、不系统讲浮点、不讲 `==`/条件、断点仅扩展档。
5. 达成证据五条是否都能在课堂收集；小结四题与参考答案是否足够检验。
6. 与 M1（固定字符串程序）和 M3（条件判断）的接口是否干净。
