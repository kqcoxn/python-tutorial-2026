# M0 · 课程介绍与编程概述 · 教师教案

> 状态：教师审核通过（2026-07-22）  
> 建议时长：30 分钟  
> 最低时长：20 分钟  
> 扩展上限：40 分钟  
> 前置要求：无  
> 后续衔接：M1 · 运行环境与第一个程序

## 1. 模块定位

M0 是整门课程的开篇，不承担系统语法教学，也不要求学生当场完成环境配置。正式进入内容前，教师需要先完成自我介绍、团队介绍和课程安排说明，让学生知道“谁会陪我学、六天怎样推进、遇到问题怎样获得帮助”。随后帮助学生建立三个坐标：

1. 编程可以出现在哪些学习、研究和生活场景中。
2. 学习编程实际包含哪些能力，而不只是记忆语言语法。
3. 本课程为什么选择 Python、控制台程序和基础问题求解作为入口。

本节的核心成果不是“知道 Python 的特点”，而是学生开始区分现实问题、算法和程序，并能把一个简单任务拆成输入、处理和输出。

## 2. 学习目标

### 2.1 知识目标

学生能够：

1. 说出至少三个不同的编程应用方向。
2. 说明程序如何帮助 AI 连接数据与工具、执行复杂任务，以及为什么结果仍需人检查。
3. 知道学习编程还包括问题分析、算法、数据表示、工具、调试、测试和协作。
4. 说明本课程选择 Python 和控制台程序作为入口的直接原因。
5. 区分现实问题、算法和程序。
6. 识别一个简单任务的输入、处理、输出和基本约束。

### 2.2 思维目标

学生能够：

1. 发现模糊任务中没有说明的条件。
2. 把“我知道怎么做”改写成别人可以逐步执行的说明。
3. 使用具体例子检查步骤是否明确、是否会停止、是否得到预期结果。

## 3. 达成证据

本节至少收集三类课堂证据：

- 学生能为一个编程应用场景说明程序可能接收什么信息、产生什么结果。
- 学生能用自己的话说明 AI、脚本和人的验证在复杂任务中的不同作用。
- 学生能执行“相邻比较排序”的无代码步骤，并指出模糊版本缺少的条件。
- 学生能通过小结题区分问题、算法和程序，并正确写出一个场景的输入、处理、输出和约束。

不以学生记住应用分类、Python 特点或排序名称作为达成标准。

## 4. 教学边界

### 本节必须完成

- 主讲、助教和六天课程安排的简要介绍。
- 编程应用与能力构成的全景图，以及程序在 AI 工作流中的执行作用。
- “本课程学什么、暂时不学什么、为什么这样选择”的定位说明。
- 一次不使用代码的精确指令活动。
- 一次输入—处理—输出拆解。
- 一个极短程序的运行结果预测。

### 本节不展开

- 各应用方向的技术栈、职业路线和语言比较。
- Python 历史、安装步骤和编辑器功能。
- `print()`、字符串、括号和引号的正式语法。
- 后续模块的语法名词清单。
- Web、移动应用、人工智能模型、第三方库和大型工程开发。

“不展开”表示暂时移出本课程主线，不表示这些内容不重要。

## 5. 课前准备

- 教师机能够运行一个只含 `print()` 的 Python 文件。
- 核对团队归属、导师、主讲经历、近期荣誉和开源贡献的展示信息与称谓。
- 准备主讲与助教信息、六天课程路线，以及可投影的应用全景图、课程边界、问题—算法—程序、输入—处理—输出和程序预测页面。
- 每两名学生准备一张可记录数字轮次变化的纸，或在学生自己的纸上完成。
- 预留板书区域，分别记录“应用场景”“模糊条件”和“输入—处理—输出”。
- 本节不集中处理安装问题；设备异常先记录，M1 再统一处理或切换备用环境。

## 6. 30 分钟建议流程

| 时间 | 环节 | 教师动作 | 学生活动 | 现场检查 |
| ---: | --- | --- | --- | --- |
| 0–4 分钟 | 认识团队与课程 | 用两页介绍中心、导师、人工智能交叉组和授课成员，再重点介绍主讲的教学经验、近期荣誉与开源贡献；随后展示六天路线、课堂节奏和求助方式。 | 认识授课团队，建立对课程范围与节奏的预期。 | 学生是否知道团队归属、课程总时长、主要路线，以及遇到问题先提供哪些信息。 |
| 4–7 分钟 | 开场问题 | 提问：“未来四年里，你觉得自己会在什么场景用到程序？AI 已经会写代码，我们为什么还要学？” | 独立思考 20 秒，再说出一个具体场景或判断。 | 回答是否超越“做软件”，并开始区分 AI 生成内容与程序执行任务。 |
| 7–11 分钟 | 编程应用与 AI 时代 | 将回答归入六个方向，说明程序也是 AI 读取文件、处理数据、调用工具和完成复杂任务的执行基础。 | 把自己的例子放入一个或多个方向，并说明其中的程序实际执行什么。 | 学生能否说出“AI 负责什么、脚本负责什么、结果怎样检查”。 |
| 11–14 分钟 | 学什么与课程定位 | 展示“语法只是其中一层”，再说明为什么用 Python、为什么从控制台开始、课程暂缓什么。 | 标记一项与原先预期不同的内容。 | 是否理解 AI 降低代码生成门槛，但没有取消问题定义、验证和调试。 |
| 14–20 分钟 | 问题、算法、程序 | 给出“把 3、1、4、1 排好”的模糊任务；追问缺失条件，再组织相邻比较活动。 | 两人一组执行精确步骤，记录每轮结果。 | 能否得到 `1、1、3、4`，并说明停止条件。 |
| 20–24 分钟 | 输入—处理—输出 | 用“小车行驶 12 米、用时 3 秒，求平均速度”拆解，再快速替换一个新场景。 | 写出输入、处理、输出和至少一个约束。 | 是否把单位和“时间不能为 0”视为问题的一部分。 |
| 24–27 分钟 | 第一个程序预测 | 先展示一行代码，不立即运行；收集预测后运行，再修改字符串并重新运行。 | 先写预测，再观察代码变化与输出变化。 | 能否把“修改—运行—观察”连成反馈闭环。 |
| 27–30 分钟 | 本章小结 | 展示三个问题，要求个人完成，再与同伴互查。 | 完成小结题。 | 是否能区分三个层次并正确拆出 IPO。 |

### 7.2 开场：未来四年，哪里会用到程序

教师提问：

> 先不要说某一种语言。未来四年里，你觉得自己会在什么具体场景用到程序？程序替你处理了什么？AI 已经能生成代码，我们为什么还需要理解编程？

先给 20 秒独立思考，再收集 4–6 个回答。学生可能首先回答“做网站”“做 App”“人工智能”。不要否定，继续追问：

- 实验数据多到无法手算时怎么办？
- 每天需要重复整理文件或表格时怎么办？
- 机器人怎样根据传感器做出动作？
- 一个模型的计算结果怎样被验证？

把回答归入以下六个方向即可，不逐项展开技术细节：

| 方向 | 程序常处理的事情 | 可用例子 |
| --- | --- | --- |
| 软件与 Web | 用户操作、业务规则和信息 | 选课系统、在线协作工具 |
| 移动应用与游戏 | 输入、状态更新和交互反馈 | App、游戏、交互媒体 |
| 自动化与工具 | 重复、机械、容易出错的步骤 | 批量整理文件、生成报告 |
| 科学计算与数据 | 计算、模拟和实验数据 | 数值计算、数据清洗、实验记录 |
| 人工智能 | 数据、模型、工具调用、评估和应用集成 | 智能助手、工作流、文本与图像任务 |
| 嵌入式与硬件 | 传感器输入和设备控制 | 机器人、实验装置、智能硬件 |

#### AI 时代的补充视角

过去，人们常为重复需求开发一个固定软件。AI 时代出现了另一种越来越常见的工作方式：面对一个具体、复杂、带有个人约束的任务，人或 AI 根据当下需求临时编写或组合脚本，让脚本读取文件、整理数据、调用模型或其他工具，再把结果交还给人检查。

可以用下面的链条说明：

```text
人的目标与约束
-> AI 协助规划或生成脚本
-> 脚本连接文件、数据、模型、API 和工具
-> 程序实际执行任务
-> 人检查结果、权限和边界，再决定是否继续
```

关键判断：

- AI 可以帮助生成代码，但程序仍是复杂任务真正“动手执行”的基础。
- 对许多一次性、短生命周期或高度定制的任务，临时按需求生成脚本，比等待一个固定软件更直接。
- 代码生成得越快，越需要人说清需求、识别风险、阅读关键逻辑、测试结果并判断是否可信。

不要把 AI 描述为可以无条件替人完成任何任务。文件权限、隐私、错误结果、外部操作和不可逆影响都需要人明确边界并验证。

收束句：

> 这些程序看起来完全不同，但都要求我们把输入、状态、规则和输出说清楚。AI 让代码更容易被生成，却让“提出正确问题、检查程序行为”变得更加重要。

### 7.3 学习编程不只是学语法

建议用六层结构说明课程关注的能力：

1. **语言语法**：怎样把指令写成合法代码。
2. **问题分析**：目标、输入、输出、规则和约束是什么。
3. **算法**：用什么有限步骤完成任务。
4. **数据表示**：程序需要记住什么，怎样组织。
5. **工具与质量**：怎样运行、调试、测试和验证。
6. **协作与表达**：怎样让别人理解、复用和改进方案。

教师讲法：

> 语法让计算机能够读懂代码，但它不能替我们决定问题有没有说清楚、步骤是否正确、结果是否可信。AI 可以帮助写出语法正确的代码，也不能自动保证需求、权限和结果正确。后面每学一个语法工具，我们都会把它放回完整的问题求解过程里。

此处不要提前解释变量、分支、循环、函数等术语。

### 7.4 为什么选择 Python 和控制台程序

**为什么使用 Python**

- 表达相对清晰，初学时可以把注意力留在问题和执行过程上。
- 修改后可以快速运行，适合小步尝试和立即验证。
- 在自动化、科学计算、数据和人工智能等大学场景中广泛使用，也适合按具体需求快速编写脚本、连接工具和验证想法。

**为什么先从控制台程序开始**

- 输入、步骤和输出直接可见，容易追踪。
- 暂时移开界面、网络和框架复杂度，先练稳定的基础能力。
- 同样的问题求解方法以后可以迁移到 Web、应用、科研和 AI 项目。

建议明确告诉学生：

> 这 10 小时不是把 Python 的全部功能快速过一遍，而是用一组基础工具反复练习分析、实现、测试和解释。

可以补充一句：

> AI 时代学习编程，不只是为了亲手输入每一行代码，更是为了能把任务说清楚、让 AI 生成的程序在正确边界内运行，并知道怎样验证它没有“看起来完成、实际做错”。

### 7.5 区分问题、算法和程序

给出三个定义：

- **问题**：想达到什么目标，以及输入、输出、规则和约束是什么。
- **算法**：解决问题的一组明确、有限、可执行的步骤。
- **程序**：用编程语言表达算法，使计算机能够执行的具体实现。

补充两点即可：

- 同一个问题可能有多种算法。
- 同一种算法也可以用不同编程语言实现。

不要要求学生在本节背诵定义。后面的无代码活动和本章小结用于验证理解。

### 7.6 无代码活动：把“排好”说清楚

#### 第一轮：暴露模糊

只给出一句任务：

```text
把 3、1、4、1 排好。
```

请学生指出这句话还没有说明什么。预期答案包括：

- 从小到大还是从大到小。
- 两个 `1` 是否都保留。
- 可以一次移动任意数字，还是只允许特定操作。
- 怎样判断已经完成。
- 输入为空或只有一个数字时怎样处理。

学生没有立即想到时，教师只追问，不直接公布全部答案。

#### 第二轮：执行精确版本

给出以下步骤：

1. 从最左边一对数字开始。
2. 比较相邻两个数字；如果左边大于右边，就交换它们。
3. 向右移动一格，继续比较下一对，直到最右端。这叫一轮。
4. 如果这一轮发生过交换，就回到第 1 步再做一轮。
5. 如果完整一轮没有发生交换，就停止。

学生两人一组，一人读步骤，一人移动数字，完成后交换角色检查。

预期状态：

```text
开始：3 1 4 1
第 1 轮：1 3 1 4
第 2 轮：1 1 3 4
第 3 轮：1 1 3 4  -> 没有交换，停止
```

活动后只提炼三个判断标准：

- 每一步能否在不猜测的情况下执行。
- 步骤是否会停止。
- 停止时是否得到题目要求的结果。

本节不要求记住“冒泡排序”这个名称。

### 7.7 输入—处理—输出

使用场景：小车行驶 12 米，用时 3 秒，求平均速度。

| 部分 | 内容 |
| --- | --- |
| 输入 | 距离 `12 m`、时间 `3 s` |
| 处理 | 平均速度 = 距离 ÷ 时间 |
| 输出 | `4 m/s` |
| 约束 | 单位一致；时间不能为 0；测量值存在精度 |

教师追问：

> 如果我只写“输入 12 和 3，输出 4”，还缺什么信息？

预期学生指出单位、两个输入各自代表什么、计算规则或有效范围。

再让学生任选一个大学学习场景，用一句话填写：

```text
输入：
处理：
输出：
规则或约束：
```

如果学生的“处理”只写“计算”“分析”，继续追问具体按什么规则处理。

### 7.8 第一个程序：先预测，再运行

先展示，不立即运行：

```python
print("把问题变成步骤。")
```

要求所有学生先写下预测，再邀请 1–2 人说明依据。随后运行程序，显示：

```text
把问题变成步骤。
```

只修改引号中的文字，再重新运行：

```python
print("让步骤接受检验。")
```

本环节只建立下面的最小反馈闭环：

```text
代码发生变化 -> 重新运行 -> 程序行为发生变化 -> 观察结果
```

不要在此处正式讲解 `print()`、字符串、括号和引号；这些内容留给 M1/M2。

## 8. 本章小结

要求学生独立回答，再与同伴互查：

1. “设计一个自动整理实验数据的工具”目前属于问题、算法还是程序？为什么？
2. “逐项读取数据，去掉无效值，再计算平均数”属于哪一层？还可能缺少什么？
3. 为“根据路程和时间求平均速度”写出输入、处理、输出和一个约束。

### 参考答案

1. 目前只是问题方向；还缺少明确输入、输出、规则和约束，尚未形成算法或程序。
2. 属于算法层面的步骤描述；仍需定义数据怎样读取、“无效值”怎样判断、空数据怎样处理等。
3. 输入为路程和时间；处理为路程除以时间；输出为平均速度；约束可以是单位一致或时间不能为 0。

如果多数学生把自然语言步骤直接称为“程序”，M1 开头用同一任务的自然语言版本和 Python 版本再做一次对照。如果多数学生遗漏约束，M2 的输入任务优先加入单位或无效输入讨论。

收齐小结题后切换到结尾页“从这里开始”，用一句话收束：“下一节，我们把今天形成的步骤意识带进一个真正由你创建、保存、运行和修改的 Python 程序。”结尾页保持显示即可，不再展开新知识。

## 9. 常见误区与处理

| 误区 | 处理方式 |
| --- | --- |
| “编程就是做 App 或网站。” | 用实验数据、批处理文件或传感器控制补充，但不展开技术栈。 |
| “会语法就会编程。” | 比较模糊排序要求与精确步骤；两者都没有代码，却有明显质量差异。 |
| “算法就是代码。” | 强调算法可以先用自然语言、图或表表达，程序是可执行实现。 |
| “计算机应该知道我的意思。” | 追问哪个词依赖常识或猜测，让学生补充规则。 |
| “控制台程序太简单，没有用。” | 说明控制台用于暴露输入、处理和输出，基础模型以后会迁移到复杂系统。 |
| “Python 简单，所以不需要严谨。” | 说明语言表达简洁不等于问题可以含糊；越容易写，越要主动验证。 |
| “AI 已经会写代码，所以不需要学编程。” | 区分生成代码和完成可靠任务：人仍需定义目标、数据、权限和约束，并检查脚本是否做了正确的事。 |
| “AI 生成的脚本只是临时用，不必理解或测试。” | 临时脚本同样可能读取隐私、修改文件或产生错误结论；生命周期短不会降低执行风险。 |

## 10. 时间调整

### 10.1 压缩到 20 分钟

- 团队和安排压缩到 2 分钟，但必须让学生知道总时长、求助方式和第一天目标。
- 应用全景只保留六个方向；AI 时代只保留“目标—生成脚本—执行—验证”一条链，不逐个收集案例。
- 课程定位只保留“为什么 Python、为什么控制台、暂缓什么”。
- 无代码活动直接使用四个数字，只完成一轮学生执行和停止条件讨论。
- 输入—处理—输出只使用平均速度示例。
- 必须保留程序预测和小结题，不能用教师直接公布答案替代。

### 10.2 扩展到 40 分钟

- 让小组各选一个大学学习或研究场景，补写输入、输出和约束。
- 让学生分析一个“AI 临时生成脚本整理实验文件”的场景，列出脚本需要的输入、权限、输出和验证方法。
- 比较两种排序说明，讨论哪一句仍然依赖猜测。
- 让学生为第一行程序提出一次修改并预测新输出。
- 增加“同一个问题为什么可能有多种算法”的短讨论。

扩展内容只在核心小结题已经能够通过时加入。

## 11. 助教任务

- 开场团队介绍时与学生见面，并明确各自负责的区域或求助方式。
- 开课前确认学生能看到课件并有纸笔记录，不在 M0 集中处理安装问题。
- 无代码活动中观察小组是否严格按步骤执行；只追问遗漏条件，不直接报最终序列。
- 输入—处理—输出练习中重点检查学生是否把“处理”写成可执行规则。
- 做小结时快速抽样，把“问题/算法/程序混淆”和“遗漏约束”两类情况反馈给主讲。
- 环境异常学生先登记，M1 再按备用环境方案处理。

## 12. 与 M1 的接口

M1 从本节最后建立的反馈闭环继续：学生将亲自创建、保存、运行和修改一个程序，并区分代码文件、运行动作和输出结果。M1 只解决第一个程序成功运行所需的问题，不提前系统讲解 Traceback 和错误类型；基础 Debug 仍安排在 M2 后半。

后续模块继续反复使用本节引出的主线：

```text
明确问题
-> 明确输入、输出、规则和约束
-> 构造具体例子
-> 拆分可执行步骤
-> 编写最小可运行版本
-> 追踪、测试、修改并解释
```

## 13. 本轮审核重点

请优先判断：

1. 30 分钟内的内容密度是否合适。
2. 自我介绍、团队介绍和六天安排是否简洁但足以建立预期。
3. AI 时代“程序是执行基础”的论述是否准确、有力度且不过度承诺。
4. “应用全景”是否清晰但没有过度展开。
5. 课程边界和选择理由是否有说服力。
6. 无代码排序活动是否适合作为第一次算法体验。
7. M0 与 M1、M2 的边界是否清楚。
