## 1. 文档用途

本文记录课程实践主线的主题决定、项目分工和设计边界，作为后续教师教案、PPT 内容稿、练习和参考实现的共同依据。

课程不再要求一个大型项目从第一天延续到最后一天，而采用两个前后衔接、各自完整的小项目：前半程强调程序在日常生活中的直接用途，后半程保留科研探索、数据分析和证据解释。两个项目共享同一套问题解决方法，但不共享必须持续维护的代码文件。

## 2. 双项目决定

### 2.1 生活项目：我的一周预算助手

学生为即将开始的大学生活制作一个控制台预算工具，输入一周预算和多笔支出，持续查看剩余金额、消费比例与预算状态，最后生成分类汇总和下一步建议。

该项目集中服务 M1–M4，在第 6 课结束时形成完整成果。第 7 课使用 M5 函数知识对已验证版本做一次小规模重构，让学生亲眼看到函数怎样改善一个自己已经使用过的程序。

项目强调：

- Python 可以解决当天就能理解和使用的生活问题。
- 程序中的变量对应真实状态，分支对应真实决策，循环对应重复记录。
- 预算判断不是道德评价，只反映用户设定的目标和当前数据。
- 核心版本只管理一次运行会话；文件保存作为课后扩展，不占用核心课时。

### 2.2 科研项目：群星彼岸·候选行星分析器

学生作为研究助理，使用一组教学化筛查规则分析候选行星的温度、重力、大气压力、液态水信号、辐射和观测完整性，输出“优先候选、需要更多观测或不建议继续”及理由。

该项目从第 7 课后半进入，在 M5–M8 中逐步形成函数化分析管线、候选目录、风险标签、研究报告和规则方案预演。它不再是飞船资源管理文字游戏，不包含燃料、任务时间、航行菜单或随机事件。

项目强调：

- 科研程序需要明确数据、规则、边界、证据和限制。
- 多项证据不能被一个不透明的万能分数取代。
- 程序输出的是教学化筛查结论，不是真实天文学定论。
- 深浅拷贝进入“比较两套研究规则而不污染基线数据”的真实所有权问题。

## 3. 为什么采用前后两阶段

两个项目不在每节课交替出现。课程先完成预算助手，再进入候选行星分析器：

1. 零基础阶段使用熟悉的生活问题，降低理解输入、状态和控制流程的额外负担。
2. 学生在第三天获得第一个可以独立运行的完整工具，形成明确完成感。
3. 第四天开始转换领域，检验学生能否把相同方法迁移到陌生的科研数据。
4. 后半程不再背负前三天累积代码，函数、容器和拷贝可以从清楚的数据模型重新开始。
5. 两个项目分别证明“程序有现实用途”和“程序能支持科学思考”，避免一个折中主题同时做不好两件事。

## 4. 共享的问题解决骨架

两个项目都使用以下工作方式：

```text
明确问题与用户
-> 找出输入、输出、规则、状态和边界
-> 先完成最小可运行版本
-> 使用条件和循环处理变化与重复
-> 用函数拆分独立职责
-> 根据数据关系选择容器
-> 使用正常、边界和不利情况测试
-> 解释结论、限制和改进方向
```

迁移不是把变量从 `expense` 改名为 `planet`。教师需要显式比较：

| 问题结构 | 预算助手 | 行星分析器 |
| --- | --- | --- |
| 输入 | 预算、支出金额、类别 | 行星观测字段、筛查规则 |
| 状态 | 累计支出、剩余预算、分类小计 | 候选目录、风险标签、研究报告 |
| 判断 | 正常、接近上限、用完、超支 | 优先、待补观测、不建议继续 |
| 重复 | 连续记录多笔支出 | 遍历并分析多个候选 |
| 边界 | 恰好用完、80% 提醒线 | 指标区间上下边界 |
| 方案比较 | 不同消费计划 | 基准与保守筛查规则 |

## 5. 教学边界

- 两个核心项目均使用控制台输入输出、函数和内置容器，不引入类、数据库、网络或第三方库。
- 预算助手核心版不承诺跨运行保存数据；JSON 或 CSV 存档只作为课后研究方向。
- 行星分析器保留虚构候选名称，但移除不服务于数据分析的世界观和资源系统。
- `try-except`、推导式、排序、默认参数和 `assert` 仍只作为进度超前扩展。
- 不要求一个文件展示全部课程知识；两个项目和独立预测、调试、拷贝任务共同形成成果证据。
- 每个项目必须独立运行、独立结束，并具有自己的正常、边界和不利测试。

## 6. 最低成果

### 预算助手

1. 接收一个正整数预算和多笔正整数支出。
2. 支持至少四种固定消费类别。
3. 每次记录后更新总支出、剩余预算和状态提示。
4. 用户可以主动结束，并得到总额、分类小计、平均支出和建议。
5. 正确处理恰好用完和超支边界。

### 候选行星分析器

1. 使用函数分析至少三个结构一致的候选记录。
2. 同时考虑多项指标，输出三类结果之一及理由。
3. 使用列表、字典、集合和元组表达各自适合的数据关系。
4. 比较两套筛查规则时不污染正式研究状态。
5. 生成承认证据限制的研究报告。

## 7. 与旧设计的关系

双项目方案保留原“群星彼岸”设计中的科研内核、三类筛查结论、候选数据和深拷贝案例，移除飞船航行、燃料、任务时间和完整文字游戏包装。

旧版火星基地资源调度模拟器、固定 V0–V8 演进路线和相应材料仍然失效。新的预算助手也不是旧资源调度项目的恢复：它围绕个人支出记录、预算反馈和生活决策，规模限定为前三天能够完成的单人实用工具。

## 8. 已确认与待细化

已确认：

- 采用两个前后衔接的小项目，不再采用一个六天大型项目。
- 第 1–3 天以生活预算助手为主，第 4–6 天以候选行星分析器为主。
- 第 7 课承担函数重构和领域迁移的桥梁作用。
- 科研项目移除飞船资源管理，保留可解释筛查与方案复制。
- 生活项目使用与瑞士纸白、终端绿并列的独立 PPT 主题“账本蓝”；规范见 [`课程演示文稿双风格参考.md`](课程演示文稿双风格参考.md) §7（2026-07-22 确认）。
- 科研项目使用与瑞士纸白、终端绿、账本蓝并列的独立 PPT 主题「星图青」，服务 M5–M8 行星分析器；规范见 [`课程演示文稿双风格参考.md`](课程演示文稿双风格参考.md) §8（2026-07-22 确认）。

后续模块制作时继续细化：

- 每节课学生看到的起始片段、故障版本和恢复版本。
- 预算示例金额是否使用统一数据或允许学生自定。
- 行星指标、阈值和候选数据的科学内容审核。
