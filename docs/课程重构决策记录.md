# 课程重构决策记录

> 状态：当前课程规划与后续内容建设的事实基线
> 更新日期：2026-07-22

## 1. 文档用途

本文记录本轮课程重构中已经确认的事实、教学自由度和明确废弃的旧假设。后续编写课纲、教案、讲义、练习、项目和 PPT 时，应先核对本文，避免从旧文件中恢复已经失效的设计。

## 2. 已确认事实

### 2.1 时间

- 每课时 50 分钟。
- 每天安排 2 个课时，即每天共 100 分钟；两个课时之间是否设置休息不属于当前已确认条件。
- 课程共 6 天、12 个课时，总计 600 分钟（10 小时）。

### 2.2 学生

- 学生已经高中毕业，是即将进入大学的准大一新生。
- 学生即将就读清华、北大层次的高校，大多会学习理工科。
- 课程按零编程基础设计，不假设学生已经使用过 Python 或其他编程语言。
- 学生整体学习能力和抽象推理能力较强，因此可以增加预测、推理、调试、方案比较和独立探索，但不能跳过零基础所需的操作与反馈。

### 2.3 内容范围

- 第 0 节为课程开篇的课程介绍与编程概述，安排在现有 12 个课时内。
- 课程最终需要覆盖 Python 基础语法、判断、循环、函数、形参实参、返回值、作用域、可变与不可变数据容器、对象可变性以及深浅拷贝。
- 对外课时名称和主讲、助教安排以 [`课程安排.xlsx`](课程安排.xlsx) 为准。

## 3. 教学自由度

- 对外课表是面向教师、家长和组织方的整体规划，不是内部授课脚本。
- 只要在 6 天内完成约定内容，知识点的具体顺序、时长和跨天位置均可调整。
- 课表中的“项目实践（一）（二）（三）”是内容占位，不规定实际项目数量。
- 项目可以集中完成，也可以拆分到多个模块；可以使用一个持续演进的作品，也可以使用多个互不相关的任务。
- 实践时间可以分散在讲解、跟练、独立编写、调试、重构和展示中，不必固定占用某个完整课时。
- 课程材料按知识模块组织；6 天授课计划负责组合模块和处理实际进度。

## 4. 核心教学决策

课程定位为“以 Python 为载体的编程与计算思维入门”，而不是 Python 语法清单。

以下思维能力必须贯穿全部教学过程：

1. 将问题说明为输入、输出、规则和约束。
2. 使用具体例子验证自己是否理解问题。
3. 将处理过程拆成精确、可执行的步骤。
4. 选择变量和数据结构表示需要保存的状态。
5. 按执行顺序追踪状态变化和控制流。
6. 主动考虑边界情况、终止条件和无效输入。
7. 阅读错误信息，提出原因假设，一次只验证一个修改。
8. 使用函数建立职责、接口和抽象。
9. 识别共享状态、副作用以及引用关系带来的行为。
10. 比较不同方案，并清楚解释自己的选择。

教师教案应同时写明知识目标和思维目标。学生材料应持续安排预测、追踪、编写、测试、调试和解释活动，而不是只提供语法说明与照抄示例。

### 4.1 课程蓝图审核决策

- M0 先介绍编程的主要应用方向和学习编程所包含的能力，再说明本课程选择 Python、控制台程序和基础问题求解的原因与边界。
- M1 只负责环境确认、运行模型和第一个程序，不在学生缺少语法经验时系统讲 Debug。
- M2 先讲值、变量、类型、输入输出和表达式，再引入 Traceback、常见错误类型和基础排错流程。
- 550 分钟建议内容加 50 分钟机动的时间模型通过。
- 保留函数先于系统容器的推荐顺序；函数应在容器、对象与拷贝和综合实践中持续复用。
- 主动排除项和跨模块成果证据通过。
- 六天推荐路线作为可调整基线通过，不变成固定逐课时脚本。

### 4.2 贯穿式案例主题决策

- 课程采用两个前后衔接的小项目，不再要求一个大型项目从第一天持续到最后一天。
- 第 1–3 天使用“我的一周预算助手”，强调程序在日常生活中的记录、计算、判断和反馈价值；第 4–6 天使用“群星彼岸·候选行星分析器”，保留科研数据分析、证据权衡和限制说明。
- 第 7 课先用函数重构已经完成的预算助手，再把相同的问题分解方法迁移到单星球分析函数，作为两个项目之间的桥梁。
- 候选行星项目移除飞船燃料、任务时间、航行菜单和随机事件，不再定位为资源管理文字游戏。
- 两个项目分别独立运行和结束，共同承担课程成果；主题调整不改变课程蓝图已经审核的模块边界和时间预算。
- 主题的详细设计与后续待定项见 [`贯穿式案例主题.md`](贯穿式案例主题.md)。
- 逐课时实现、成果评价和学生自由扩展的初稿见 [`贯穿式项目实施计划.md`](贯穿式项目实施计划.md)，通过教师审核前不视为最终教学脚本。

## 5. 材料建设决策

- 第 0 节保留为独立内容模块，主题是课程介绍与编程概述。
- 教师教案、练习和 PPT 内容稿以内容模块为基本建设单位，不按课时编号硬绑定。
- 第 0 节固定为课程开篇；M1–M8 采用已审核的推荐依赖顺序，具体跨天位置仍可按检查点调整。
- 推荐授课路线可以按 6 天编排，但应标注依赖关系、最低完成点、可压缩项和可移动项。
- 不单独制作学生讲义、PPT 实现或示例代码目录；必要的学生可见内容与代码示例直接进入 PPT 内容稿，练习及答案进入练习材料。
- PPT 内容稿不使用“清北苗子”等内部画像表述。
- PPT 内容稿只保留学生当场需要阅读、观察或操作的内容；教师口令、助教分工、答案和进度调整放在教师教案中。
- 双项目实践主题已经确定，详细设计以 [`贯穿式案例主题.md`](贯穿式案例主题.md) 为准。PPT 最终视觉系统尚未确定；瑞士纸白与终端绿保留为候选风格参考，像素轨道和旧三风格体系不再作为默认要求。

## 6. 已废弃假设

以下内容已经失效，不得作为后续设计依据：

- 学生是高一或初升高混班。
- 基础课程只有 3 次课、每次 2 小时或总计 6 小时。
- 课程必须按照固定 10 章顺序逐章推进。
- 现有章节与第 1–6 课时存在固定映射。
- 三次“项目实践”必须对应三个独立项目或三个完整实践课时。
- 火星基地资源调度模拟器是指定贯穿项目。
- 课程必须使用旧版 V0–V8 项目演进方式。
- 旧版三套 PPT 视觉风格和章节课件结构必须继续使用。
- 旧版 30 章、三阶段和 AI 编程内容属于本次 12 课时课程范围。

## 7. 文档优先级

后续发现信息冲突时，按以下顺序处理：

1. 本决策记录
2. `课程安排.xlsx` 中的对外内容和人员安排
3. `课程规划基线.md` 及后续新建课程设计文档

教师后续确认的新要求应先同步到本文，再用于大规模内容建设，避免只存在于对话中的隐含要求成为长期依赖。
