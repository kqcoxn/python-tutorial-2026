## 2. 主题决定

课程采用以“行星气候沙盘”为科研内核的深空宜居行星探索主题，暂定名称为：

> **群星彼岸：宜居行星探索计划**

案例表现形式为科研探索型文字游戏。学生扮演深空探测舰的宇航员兼科研官，驾驶飞船访问多个目标星系，扫描其中的候选星球，分析恒星、轨道、温度、大气、水资源、重力和辐射等观测信息，在有限条件下选择探索路线，并最终提出适合人类长期生存的宜居行星候选；证据不足时也可以作出暂不推荐的结论。

星际载人航行属于科幻叙事设定；行星宜居性判断应尽量建立在清楚、可解释的科学指标上。学生需要根据证据作出选择，而不是接受程序给出的唯一答案。

## 3. 核心体验

案例使用以下循环组织持续演进的任务：

```text
接收候选星球
-> 扫描并取得观测信息
-> 分析宜居条件与风险
-> 选择航线或后续行动
-> 处理航行与科研事件
-> 标记为优先候选、建议继续观测或放弃
-> 更新任务状态与研究档案
-> 提交最终候选及证据说明
```

案例同时包含三个层次：

1. **冒险层**：航行、扫描、目标选择、未知事件和最终发现提供叙事驱动力。
2. **策略层**：燃料、时间和设备状态等有限条件使选择产生后果。
3. **科研层**：恒星与行星观测指标、证据权衡、边界检查和结论解释构成任务核心。

冒险和资源管理用于制造问题与选择，不能取代科研分析。课程成果不以“剩余燃料最多”作为主要标准，而应考查学生能否正确处理数据、解释判断依据并说明方案限制。

登陆和突发事件可以作为叙事或玩法扩展，但不属于核心实现要求；核心终局是提交宜居候选或作出“当前不推荐任何候选”的研究结论。

## 4. 教学作用

贯穿式案例为课程提供统一语境，但不改变 [`curriculum-map.md`](curriculum-map.md) 中已经审核的知识范围、依赖关系和时间模型。

后续设计遵循以下原则：

- Python 知识应对应游戏中的真实行动，例如计算、判断、扫描、航行、记录、复核和生成报告，避免只替换变量名的主题包装。
- 主要案例程序可以从最小可运行的文字输出逐步演进，最终能够维护飞船状态、分析多个星球并支持玩家决策；它与主题内的独立短任务共同组成课程成果。
- 条件与循环承担探索和事件推进，函数承担职责拆分，容器承担飞船、星球、任务和研究档案的数据建模。
- 对象可变性与深浅拷贝应进入有实际所有权问题的场景，例如任务快照、平行航线推演或独立科研方案，不能作为脱离情境的附加剧情。
- 不要求每个语法点都进入同一个最终程序。预测、追踪、Debug 和深浅拷贝等能力可以由主题内的独立短任务提供证据。
- 核心版本保持控制台交互，不以图形界面、第三方库、文件读写、数据库或网络访问为前提。

## 5. 科学与科幻边界

- 允许使用虚构飞船、星际航行技术、任务组织和候选星系，建立完整的探索叙事。
- 恒星类型、宜居带、表面温度、大气、液态水、重力和辐射等概念应采用经过核查的简化模型。
- “宜居”应表现为多项证据之间的权衡，不设置无需理解即可直接采用的万能分数。
- 不把一次扫描写成确定发现生命；程序给出的是宜居候选、风险和后续建议。
- 资源消耗与突发事件必须服务于问题求解，避免重新变成以资源调度为主体的旧版火星基地项目。

## 6. 与旧版项目的关系

本主题是重新设计的贯穿式案例，不恢复旧版火星基地资源调度模拟器、固定 V0–V8 演进路线或旧项目材料。

两者都使用太空探索叙事，但核心任务不同：旧版以基地资源调度为中心；新主题以宜居行星观测、科学分析和探索决策为中心。旧版教案、课件、代码和测试仍然不具有参考优先级。
