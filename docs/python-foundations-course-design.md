## 1. 文档目的

本文档是后续章节教案、课件、示例代码、练习题和环境材料的总设计依据。课程按知识章节准备，不强制绑定具体课时。教师可以根据课堂进度在章节边界暂停或继续，但应保持知识依赖关系和贯穿项目的连续性。

本阶段暂定为 3 次课，每次 2 小时，共 6 小时。6 小时是较紧的安排，尤其是学生设备和操作系统不统一。因此，软件安装应尽量作为课前任务完成，第一章课堂只进行环境验证和问题救援。

## 3. 课程定位与最终能力

### 3.1 课程定位

本阶段不是 Python 语法百科，也不追求提前覆盖后续的数据结构和函数内容。核心任务是帮助学生建立第一个完整的编程闭环：

```text
理解问题 -> 设计步骤 -> 编写代码 -> 运行观察 -> 阅读错误 -> 修改验证
```

### 3.2 最终能力

完成本阶段后，学生应能够：

1. 独立创建、保存并运行一个 Python 控制台程序。
2. 使用 `print()`、变量、数值、字符串和格式化输出表达程序状态。
3. 使用 `input()` 获取用户输入，并进行必要的类型转换。
4. 使用算术、赋值、比较和逻辑运算符构造表达式。
5. 使用 `if`、`elif`、`else` 和嵌套判断表达分支逻辑。
6. 使用 `while` 和 `for` 实现重复执行，理解循环条件和次数。
7. 在合适的位置使用 `break` 和 `continue`。
8. 阅读常见 Python 错误信息，并通过断点或临时输出定位问题。
9. 独立完成并讲解一个包含输入、分支和循环的火星基地资源调度模拟程序。

### 3.3 内容边界

本阶段原则上不正式讲授以下内容：

- 列表、元组、字典和集合
- 自定义函数和 `return`
- 文件读写
- 面向对象
- 第三方库

贯穿项目和练习不应依赖学生尚未学习的知识。教师演示中如必须使用超纲代码，应将其封装为不可修改的脚手架，并明确说明暂不要求理解。

## 4. 总体教学原则

### 4.1 以可运行作品为主线

所有章节共同扩展一个“火星基地资源调度模拟器”。每学完一个知识点，学生都能看到模拟器新增的具体能力，避免章节示例彼此割裂。

### 4.2 Debug 贯穿全程

Debug 不作为讲完即结束的孤立章节。第一章建立基本排错流程，后续每章固定加入真实的错误案例和找错任务。

统一使用下面的排错流程：

```text
停下来 -> 读报错最后一行 -> 找错误类型 -> 找行号
-> 检查该行及上一行 -> 提出一个原因 -> 只改一处 -> 重新运行
```

遇到逻辑错误时，再使用断点、单步执行或临时 `print()` 查看变量变化。

### 4.3 一次只增加一个主要难点

早期示例控制在 5 到 15 行。讲解新概念时，其他部分应尽量保持简单，不要在同一个例子中同时引入多个新语法。

### 4.4 先预测，再运行

学生运行代码前，先回答以下问题之一：

- 会输出什么？
- 哪一行会先执行？
- 变量此时是什么值？
- 循环会执行几次？
- 程序会不会结束？

这类问题适合学习能力较强的学生，也能防止课堂退化为机械抄写。

### 4.5 核心任务与挑战任务分层

每章采用约 80% 共同核心和 20% 扩展挑战：

- 核心任务：所有学生都应完成，直接对应最终能力。
- 挑战任务：提供更复杂的边界条件、多种解法、嵌套逻辑或调试要求。

扩展任务不能引入尚未讲授的数据结构和函数，以免快慢学生进入不同的知识体系。

## 5. 建议的章节架构

下面的顺序对原课纲进行了重组。所有原定知识点仍然覆盖，但依赖关系更清晰。

### 第 1 章：环境、Python 与第一个程序

**核心目标**

- 知道 Python 是什么以及代码如何被执行。
- 完成 Python 和 PyCharm 的安装、解释器配置与运行验证。
- 能够创建 `.py` 文件并运行第一个程序。

**核心知识**

- Python、解释器、代码文件和 IDE 的关系
- PyCharm 项目、Python 解释器和运行按钮
- `print()` 的首次使用
- 文件保存位置和 `.py` 扩展名

**贯穿项目增量**

显示任务标题、火星基地背景和任务简报。

**典型问题**

- Python 安装成功但 PyCharm 没有选择解释器
- Windows 中 `python` 命令不可用，但 `py` 可用
- macOS 中需要使用 `python3`
- 文件被保存为 `.py.txt`
- 运行了错误的文件或错误的运行配置

**最低完成标准**

学生能够独立创建并运行：

```python
print("火星基地资源调度系统启动！")
```

### 第 2 章：输出、注释、错误信息与基础 Debug

**核心目标**

- 使用 `print()` 输出多个内容。
- 区分注释、程序代码和运行结果。
- 认识语法错误与运行时错误，掌握基础排错流程。
- 会设置断点、单步执行和查看变量。

**核心知识**

- 单行注释 `#`
- 多次输出和 `print()` 的多个参数
- Traceback 的阅读顺序
- `SyntaxError`、`IndentationError`、`NameError`
- 断点、单步执行、继续运行

**贯穿项目增量**

为任务简报代码增加注释，并调试教师提供的故障版本。

**教学提醒**

注释用于说明目的和关键决策，不要求给每一行代码写中文翻译。Debug 演示应先预测变量变化，再展示工具按钮。

### 第 3 章：变量、标识符与基本类型

**核心目标**

- 理解变量用于保存程序状态。
- 能够创建、读取和修改变量。
- 认识整数、浮点数、字符串和布尔值。

**核心知识**

- 赋值语句
- `int`、`float`、`str`、`bool`
- `type()` 作为观察工具
- 标识符规则和有意义的变量名
- 推荐使用 `snake_case`

**贯穿项目增量**

增加基地名称、任务日、能源、氧气、饮用水、科研积分和系统状态：

```python
base_name = "星火基地"
mission_day = 1
energy = 100
oxygen = 100
water = 100
research_points = 0
system_online = True
```

**典型问题**

- 变量先使用后定义
- 大小写不一致
- 变量名中包含空格或以数字开头
- 将 `=` 误认为数学等号，而忽略“把右侧结果保存到左侧”的执行过程

### 第 4 章：字符串、格式化输出与转义字符

**核心目标**

- 创建和拼接字符串。
- 使用 f-string 输出包含变量的完整信息。
- 正确处理引号、换行和制表符。

**核心知识**

- 单引号和双引号
- 字符串拼接的基本概念
- f-string
- `\n`、`\t`、`\\`、`\"`、`\'`

**贯穿项目增量**

生成基地状态面板和多行任务简报：

```python
print(f"基地：{base_name}  任务日：{mission_day}")
print(f"能源：{energy}  氧气：{oxygen}  饮用水：{water}")
print("今日任务：\n1. 维护供能系统\n2. 维护生命保障系统\n3. 开展科研探索")
```

**典型问题**

- 引号没有成对出现
- f-string 前遗漏 `f`
- 数字和字符串直接使用 `+` 拼接导致 `TypeError`
- Windows 路径中的反斜杠被解释为转义字符

### 第 5 章：输入、类型转换、算术与赋值运算

**核心目标**

- 使用 `input()` 获取用户输入。
- 理解 `input()` 的结果始终是字符串。
- 将输入转换为数字并参与计算。
- 使用算术和复合赋值更新基地资源。

**核心知识**

- `input()`
- `int()`、`float()`、`str()`
- `+`、`-`、`*`、`/`、`//`、`%`、`**`
- `+=`、`-=`、`*=`、`/=`
- 运算顺序和括号

**贯穿项目增量**

允许学生为基地命名、输入任务选择并计算资源消耗：

```python
base_name = input("请输入火星基地名称：")
energy_cost = int(input("本次任务预计消耗多少能源？"))
energy -= energy_cost
print(f"{base_name} 当前剩余能源：{energy}")
```

**典型问题**

- 忘记将数字输入转换为 `int`
- 输入非数字导致 `ValueError`
- `/` 与 `//` 混淆
- `=`、`+=` 和 `==` 混淆

**安全边界**

不要把 `eval()` 作为输入转换工具。它会执行输入内容，不适合零基础课程，也会形成不安全的编程习惯。

### 第 6 章：布尔值、比较运算与逻辑运算

**核心目标**

- 理解条件表达式的结果是 `True` 或 `False`。
- 能够组合多个条件。
- 能够口头解释复杂条件何时成立。

**核心知识**

- `==`、`!=`、`>`、`<`、`>=`、`<=`
- `and`、`or`、`not`
- 真值表和短路求值的直观认识

**贯穿项目增量**

判断基地是否满足开展科研探索的条件：

```python
can_explore = energy >= 20 and oxygen >= 15 and water >= 10
```

**挑战任务**

给出多个条件表达式，让学生在不运行代码的情况下判断结果，并尝试构造能让表达式成立的变量值。

### 第 7 章：`if`、`elif`、`else` 与嵌套判断

**核心目标**

- 使用分支结构让程序根据输入做出不同反应。
- 理解多个独立 `if` 与 `if-elif-else` 的区别。
- 在必要时使用嵌套判断，但避免无意义的深层嵌套。

**核心知识**

- `if`
- `if-else`
- `if-elif-else`
- 缩进与代码块
- 嵌套判断
- 条件覆盖和分支顺序

**贯穿项目增量**

实现维护供能、维护生命保障和科研探索等不同决策，并根据资源条件给出不同结果。

**典型问题**

- 条件结尾缺少冒号
- 缩进不一致
- 使用 `=` 代替 `==`
- `elif` 顺序导致更具体的条件永远无法执行
- 多个 `if` 导致本来只应选择一个的分支被重复执行

**挑战任务**

- 找出无法到达的分支。
- 用逻辑运算减少一层嵌套。
- 为边界值设计测试用例。

### 第 8 章：`while` 循环与循环嵌套

**核心目标**

- 理解“条件成立时重复执行”。
- 识别循环变量、循环条件和状态更新。
- 能够发现并修复常见死循环。
- 理解嵌套循环的执行次数。

**核心知识**

- `while`
- 计数器和状态更新
- 条件控制循环
- 死循环
- 循环嵌套

**贯穿项目增量**

让学生在输入无效时重新选择，并在任务天数和关键资源满足条件时继续运行基地。

**典型问题**

- 忘记更新循环条件相关变量
- 更新方向错误
- 循环条件永远为真或一开始就为假
- 嵌套循环中的变量更新位置错误

**挑战任务**

不运行代码，计算内外层循环分别执行多少次，并解释每次变量值。

### 第 9 章：`for`、`range()`、`break` 与 `continue`

**核心目标**

- 使用 `for` 完成固定次数的重复任务。
- 理解 `range()` 的开始、结束和步长。
- 使用 `break` 提前结束循环。
- 使用 `continue` 跳过本轮剩余代码。

**核心知识**

- `for`
- `range(stop)`、`range(start, stop)`、`range(start, stop, step)`
- 左闭右开的范围
- `break`
- `continue`
- `while` 与 `for` 的选择

**贯穿项目增量**

实现任务倒计时、固定天数模拟、通信重试，并在资源耗尽或任务完成时退出循环。

**典型问题**

- 误以为 `range(1, 5)` 包含 5
- 在 `while` 中使用 `continue` 前没有更新必要状态，造成死循环
- 把 `break` 误认为退出整个程序
- 在嵌套循环中误判 `break` 退出的层级

**挑战任务**

分别使用 `while` 和 `for` 解决同一问题，比较哪一种更符合题意。

### 第 10 章：综合作业、自由完善与展示

**核心目标**

- 综合使用本阶段知识完成可运行作品。
- 根据测试结果自行修复问题。
- 能够用简洁语言解释程序结构和一个关键设计。

建议在最后一次课预留：

- 30 分钟：学生自由完善作品，教师和助教巡回支持。
- 15 到 25 分钟：4 到 6 位学生自愿展示，每人约 3 分钟。

若实际进度不足，优先保证核心项目可运行，缩减挑战功能和展示人数，不压缩学生独立编写与调试的时间。

## 6. 贯穿项目：火星基地资源调度模拟器

### 6.1 项目定位

学生扮演火星基地任务控制员，在 7 个任务日内管理能源、氧气和饮用水，通过维护与科研决策保证基地持续运行，并尽可能获得科研积分。

项目定位为“简化的工程决策模拟器”，而不是单纯的冒险游戏。学生需要把任务状态抽象为变量，用运算更新资源，用条件判断决策是否可行，再用循环推进任务周期。家长和课程管理者可以从最终成果中直接看到问题建模、逻辑决策、边界检查和程序调试能力。

项目不依赖图片、第三方库、列表或函数。它只使用本阶段知识，既可以在 PyCharm 中运行，也可以复制到在线 Python 环境中运行。

### 6.2 科学与表达边界

本项目是为编程教学设计的规则模型，不是航天工程仿真软件。能源、氧气、饮用水和科研积分均使用无单位的教学数值，不宣称对应真实火星基地参数。

备课和展示时遵循以下原则：

- 将资源变化称为“模拟规则”，不称为真实消耗数据。
- 核心版不依赖随机事件，确保每次输入都可以复现和调试。
- 资源耗尽时使用“启动应急撤离，任务结束”，避免不必要的伤亡叙事。
- 不使用“AI 调度”或“智能决策”等名称，因为本阶段程序没有使用 AI。
- 强调学生是在学习如何把现实问题简化成可计算模型。

### 6.3 核心状态模型

| 变量 | 建议初始值 | 含义 |
| --- | ---: | --- |
| `base_name` | 学生输入 | 火星基地名称 |
| `mission_day` | `1` | 当前任务日 |
| `max_days` | `7` | 模拟总天数 |
| `energy` | `100` | 基地可用能源 |
| `oxygen` | `100` | 生命保障系统中的氧气资源 |
| `water` | `100` | 基地可用饮用水资源 |
| `research_points` | `0` | 科研任务成果积分 |
| `mission_active` | `True` | 任务是否仍在进行 |

核心版只保留这些变量。后续如增加通信状态、设备可靠度或乘员状态，应作为挑战任务，避免基础版过度复杂。

### 6.4 核心决策规则

每个任务日执行一次“显示状态 -> 输入决策 -> 更新资源 -> 扣除基础消耗 -> 检查任务状态”的循环。

建议使用下面的教学默认值：

| 决策 | 直接效果 |
| --- | --- |
| 维护供能系统 | 能源 `+25` |
| 维护生命保障系统 | 能源 `-15`，氧气 `+20`，饮用水 `+15` |
| 开展科研探索 | 能源 `-20`，氧气 `-10`，饮用水 `-8`，科研积分 `+30` |

每个有效任务日结束后，再扣除基地基础消耗：能源 `-5`、氧气 `-10`、饮用水 `-8`。资源恢复后超过 `100` 时，将其限制为 `100`。

这些数值是第一版教学参数。制作最终参考实现后，应实际运行多种决策组合，确保学生既可能完成任务，也可能因不合理调度触发应急撤离。

任务结束时按以下规则给出评价：

| 条件 | 任务评价 |
| --- | --- |
| 任一关键资源小于等于 `0` | 启动应急撤离，任务提前结束 |
| 完成 7 天且科研积分大于等于 `90` | 卓越科研任务 |
| 完成 7 天且科研积分大于等于 `60` | 火星任务顺利完成 |
| 完成 7 天但科研积分不足 `60` | 基地稳定运行，科研目标未完全达成 |

对 7 天内所有 `3^7 = 2187` 种决策组合进行初步枚举后，四类结果都可以实际到达。350 种卓越科研任务路径全部至少包含一次生命保障维护，因此三类核心决策都有实际用途。可使用以下代表性序列作为后续参考实现的测试基线，其中 `1`、`2`、`3` 分别表示三种核心决策：

| 决策序列 | 预期结果 |
| --- | --- |
| `1-1-1-1-1-1-1` | 基地稳定运行，但科研积分不足 |
| `3-3-1-1-1-1-1` | 火星任务顺利完成 |
| `3-3-3-2-1-1-1` | 卓越科研任务 |
| `3-3-3-3` | 能源耗尽，启动应急撤离 |

### 6.6 最低验收要求

最终作品至少应包含：

- 一个清晰的系统标题和任务目标
- 由学生输入的基地名称或任务选择
- 至少五个有意义的状态变量
- 数值、字符串和布尔值的使用
- 使用 f-string 显示资源状态面板
- 使用算术或复合赋值更新资源
- 至少一个 `if-elif-else` 决策结构
- 至少一个循环推进任务或验证输入
- 合理使用至少一个 `break` 或 `continue`
- 同时包含任务完成和应急撤离两类结果
- 能够在教师给定的测试输入下正常结束，不出现无法解释的死循环

### 6.7 挑战要求

进度较快的学生可以选择：

- 同时使用 `while` 和 `for`
- 使用外层任务日循环和内层输入验证循环
- 同时合理使用 `break` 与 `continue`
- 增加通信状态、设备可靠度或基地维护成本
- 增加第四种调度决策，但说明它对各资源的影响
- 设计由任务日或资源条件触发的可复现事件
- 为卓越、完成、未达标和应急撤离分别准备测试用例
- 比较两套调度策略，并解释哪一套更稳定

### 6.8 展示要求

每位展示者控制在约 3 分钟：

1. 30 秒介绍基地名称、任务目标和自己采用的调度策略。
2. 60 秒运行程序，展示两次决策和资源面板变化。
3. 60 秒解释一个条件判断或循环如何控制任务流程。
4. 30 秒说明遇到的一个错误、使用的测试输入及修复方式。

### 6.9 成果表达口径

对外展示时，建议学生使用下面的表达方式：

> 我设计了一个简化的火星基地资源调度模型。程序使用变量保存能源、氧气、饮用水和科研积分，通过条件判断处理不同决策，再用循环模拟连续 7 个任务日。我还测试了资源临界值和无效输入，确保程序能够正常完成任务或进入应急撤离流程。

重点展示“如何建模、如何判断、如何循环、如何测试”，而不是只介绍故事情节。

## 7. 每章统一备课模板

后续编写每章教案时，统一包含以下部分：

1. **章节目标**：使用“学生能够……”描述，不使用“了解……”作为唯一目标。
2. **开始条件**：学生需要已经掌握的知识和已经完成的项目版本。
3. **核心词汇**：本章最多 5 到 8 个新术语。
4. **最小示例**：只展示本章最核心的新知识。
5. **贯穿项目增量**：在上一版本上增加一个明确功能。
6. **跟随练习**：学生在教师带领下完成。
7. **独立练习**：学生脱离逐行提示完成。
8. **找错任务**：包含真实且有代表性的错误。
9. **挑战任务**：面向进度较快的学生。
10. **检查点**：用一个可观察结果判断是否可以进入下一章。
11. **教师提示**：常见误解、提问方式和时间不足时的删减项。
12. **参考答案**：核心版与挑战版分开保存。

## 10. 环境与网络预案

### 10.1 环境路径优先级

按以下顺序组织课堂环境：

1. **主路径：本地 Python + PyCharm**  
   功能完整，适合讲解项目、断点和单步调试。
2. **备用路径 A：本地 Python + IDLE**  
   Python 已安装但 PyCharm 无法安装或配置时使用。IDLE 随官方 Python 安装，能够完成本阶段的大部分代码练习。
3. **备用路径 B：在线 Python 环境**  
   学生能够联网，但本地环境暂时不可用时使用。
4. **最后保障：同伴结对或备用电脑**  
   学生既无法安装又无法联网时使用。U 盘不能解决管理员权限、系统限制或硬件故障。

### 10.2 在线环境建议

建议使用 [Online Python](https://www.online-python.com/) 作为临时备用环境。

截至 2026-07-18，网站可以直接打开 Python 编辑器，没有前置登录页面；页面标明免费使用、支持 Python 3.12、实时交互输入和在线 Debug。正式上课前仍需在教学地点网络中重新测试访问速度，并运行包含 `input()` 的示例。

在线环境只用于临时救场，原因包括：

- 学生网络并不确定。
- 网站功能和访问策略可能变化。
- 浏览器页面布局与 PyCharm 不同。
- 在线环境不适合承担本地解释器和项目文件管理的全部教学目标。

### 10.3 课前安装策略

建议在正式上课前 3 到 7 天发放安装说明，要求学生提交以下任一验证截图：

- PyCharm 成功运行 `print("Hello, Python!")`
- 本地 IDLE 成功运行相同代码
- 明确报告无法安装的系统和错误信息

不要只询问“是否安装完成”，要以实际运行结果作为判断标准。

## 11. 主讲教师与助教分工

### 主讲教师

- 控制全班节奏和章节检查点。
- 演示核心操作、思考过程和 Debug 方法。
- 观察共性问题，并在全班统一纠正。
- 决定继续下一章还是增加练习。

### 助教

- 第一阶段优先处理安装、解释器和运行配置问题。
- 记录重复出现的问题，及时反馈给主讲教师。
- 使用提问帮助学生定位问题，不直接替学生完成代码。
- 关注已经停止操作但没有主动求助的学生。
- 在项目完善阶段检查学生是否满足最低验收要求。

### 建议的求助流程

学生求助前先完成：

1. 重新阅读错误信息最后一行。
2. 指出错误行号。
3. 说出自己认为的一个可能原因。
4. 向同桌说明当前卡点。
5. 仍未解决时再请求助教。

## 12. 常见错误教学清单

| 类型 | 常见表现 | 建议教学处理 |
| --- | --- | --- |
| `SyntaxError` | 引号、括号、冒号缺失 | 先看错误行及上一行，检查符号是否成对 |
| `IndentationError` | 代码块缩进不一致 | 打开空白字符显示，统一使用 4 个空格 |
| `NameError` | 变量未定义或拼写错误 | 对照变量首次赋值位置和大小写 |
| `TypeError` | 字符串与数字直接运算 | 先打印值和 `type()`，再决定转换方向 |
| `ValueError` | `int()` 接收到非数字文本 | 检查真实输入，设计有效输入范围 |
| 逻辑错误 | 程序运行但结果错误 | 预测变量值，使用断点或临时输出逐步验证 |
| 死循环 | 程序持续运行无法结束 | 找循环条件和相关变量的更新语句 |
| 边界错误 | `range()` 少一次或多一次 | 写出实际生成的数字序列再判断 |
| 运行配置错误 | 修改代码后输出没有变化 | 确认运行的文件和当前编辑文件一致 |

### 13.2 最终项目评价建议

| 维度 | 比例 | 判断依据 |
| --- | ---: | --- |
| 可运行性 | 20% | 程序能够启动、交互并正常结束 |

## 14. 进度调整规则

### 必须保证

- 学生能够运行代码
- 变量、输入输出和类型转换
- 比较、逻辑运算与完整分支
- 至少一种循环的独立使用
- 最终项目的独立编写和调试时间

### 可以压缩

- Python 历史和应用领域介绍
- 重复性的教师演示
- 复杂格式化细节
- 深层嵌套练习
- `while` 与 `for` 的多种等价写法

### 可以转为挑战任务

- 多层嵌套循环
- 复杂短路求值分析
- 多个边界条件的系统测试
- 同一问题的多种算法比较
- 同时使用 `break` 和 `continue` 的复杂流程

如果进度明显落后，不要通过取消独立练习来追赶。应减少扩展内容和演示数量，保留学生亲自编写、运行和调试的时间。
