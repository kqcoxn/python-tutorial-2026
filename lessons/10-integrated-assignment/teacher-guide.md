# 第 10 章教师教案：综合作业、自由完善与展示

> 更新日期：2026-07-18  
> 建议用时：80-120 分钟  
> 项目版本：V8 到学生 Final

## 1. 章节定位

本章不讲新语法，整章就是贯穿项目的集中完成与展示阶段。学生要把“能运行一次”提升为“经过测试、可以解释、适合展示的作品”。教师评价重点是建模、逻辑、调试和表达，不是故事包装或功能数量。

## 2. 学习目标与验收证据

| 学习目标 | 验收证据 |
| --- | --- |
| 综合使用本阶段知识 | 最终程序包含输入、变量、运算、分支和循环 |
| 使用测试发现并修复问题 | 提交至少四组测试结果和一条 Debug 记录 |
| 在语法边界内自主完善 | 新功能可复现、能结束、学生能够解释 |
| 说明一个关键设计 | 能追踪一个条件或循环如何影响状态 |
| 进行短时成果展示 | 3 分钟内完成介绍、运行、解释和复盘 |

最低标准：核心程序能够完成 7 个有效任务日或进入应急撤离；无效输入不会无故消耗任务日；学生能解释一处条件与一处循环。

## 3. 开始条件与交付物

学生应完成 V8 或拥有等价的可运行版本。最终提交：

1. `mars_base_final.py`。
2. 学生讲义中的测试记录。
3. 一条错误、原因、修改和复测记录。
4. 一张 3 分钟展示提纲。

不要求提交真实航天数据、图片、幻灯片或额外文档。程序必须保持本阶段语法边界：不使用导入、列表、字典、函数、类、文件、异常处理或第三方库。

## 4. 教学流程与时间安排

| 模块 | 建议用时 | 教师动作 | 学生产出 |
| --- | ---: | --- | --- |
| A. 基线冻结 | 8-12 分钟 | 宣布规则、复制 Final 文件 | 可运行基线 |
| B. 核心验收 | 15-20 分钟 | 分配测试序列 | 测试矩阵 |
| C. Debug 修复 | 10-15 分钟 | 巡查共性错误 | 修复记录 |
| D. 自由完善 | 30 分钟 | 只提供边界和追问 | 一个可解释改进 |
| E. 展示准备 | 8-12 分钟 | 提醒三分钟结构 | 展示提纲 |
| F. 自由展示 | 15-25 分钟 | 邀请 4-6 人 | 同伴反馈 |

若总时间不足，优先保留 A、B、D 的独立编写时间和至少 3 人展示。不要为了增加展示人数压缩学生自己的完善与测试。

## 5. 基线冻结

学生先复制当前可运行版本为 `mars_base_final.py`。冻结要求：

- 原 V8 保留，不在原文件上冒险修改。
- Final 文件第一次运行结果与 V8 一致。
- 教师参考答案不直接发给学生；学生从自己的版本继续。
- 如果学生 V8 尚未通过，先完成核心修复，不进入扩展功能。

主讲宣布对外口径：这是“简化的工程决策模型”，所有资源是无单位教学模拟值，不宣称是真实火星数据，也不称为 AI 调度。

## 6. 章节检查点：核心验收清单

最终作品至少包含：

- [ ] 清晰的系统标题和任务目标。
- [ ] 用户输入的基地名称或任务选择。
- [ ] 至少五个含义明确的状态变量。
- [ ] 数值、字符串和布尔值。
- [ ] f-string 状态面板。
- [ ] 算术或复合赋值更新资源。
- [ ] 至少一条 `if-elif-else` 分支链。
- [ ] 至少一个循环推进任务或验证输入。
- [ ] 至少合理使用一次 `break` 或 `continue`。
- [ ] 任务完成与应急撤离两类结果。
- [ ] 给定测试输入下可以正常结束。

核心项未通过时，助教只处理最先阻塞的行为，不替学生重写整个文件。

## 7. 必测矩阵

| 编号 | 决策序列 | 预期 |
| --- | --- | --- |
| T1 | `1 1 1 1 1 1 1` | 7/7，科研 0，稳定运行 |
| T2 | `3 3 1 1 1 1 1` | 7/7，科研 60，顺利完成 |
| T3 | `3 3 3 2 1 1 1` | 7/7，科研 90，卓越任务 |
| T4 | `3 3 3 3` | 第 4 个有效任务日能源耗尽，应急撤离 |
| T5 | 先输入 `9`，再执行任一完整序列 | `9` 不消耗任务日和资源 |
| T6 | 基地名称直接回车 | 使用默认名称“星火基地” |

课堂至少完成 T1、T3、T4、T5；T2、T6 可由同桌交叉验证。每次从新运行开始，不能沿用上一组状态。

## 8. Debug 记录要求

学生至少提交一条真实记录：

| 项目 | 要求 |
| --- | --- |
| 现象 | 实际发生了什么，不写“程序坏了” |
| 预期 | 本来应该发生什么 |
| 定位证据 | 错误类型、行号、变量值或测试序列 |
| 原因 | 一个可验证的解释 |
| 修改 | 只描述关键改动 |
| 复测 | 使用哪组输入，结果如何 |

逻辑错误示例：“输入 9 后任务日变成 2”，定位时查看 `mission_day += 1` 是否在 `continue` 之前。

## 9. 综合练习：30 分钟自由完善

学生只选一项主改进，先写规则和测试，再改代码。

### 方向 A：表达与可读性

- 自定义基地名称、任务口号和报告文字。
- 改进变量名、注释、分隔线和状态面板。
- 保持输出紧凑，不增加无意义长故事。

### 方向 B：第四种调度决策

- 明确编号、名称和四项状态变化。
- 更新合法输入判断、菜单和分支。
- 至少设计一组测试证明它不是无效装饰。
- 资源变化必须称为教学模拟规则。

### 方向 C：可复现事件

- 由固定任务日或固定资源条件触发，不使用随机数。
- 说明触发前后状态变化。
- 不造成无法结束或无法测试的流程。

### 方向 D：更好的验证与反馈

- 增加更清楚的无效输入提示。
- 显示决策前后的资源变化。
- 增加边界值提示，但不引入异常处理或容器。

拒绝标准：复制无法解释的完整代码、引入超纲语法、使用随机事件导致展示不可复现、声称项目使用了 AI 或真实航天参数。

## 10. 教师与助教分工

### 主讲

- 每 10 分钟广播一次剩余时间和当前阶段。
- 只统一处理影响 3 人以上的共性问题。
- 用追问控制扩展范围：“规则是什么？会改哪些变量？怎样测试？”
- 提前 10 分钟停止新增功能，要求保存、复测和准备展示。

### 助教

- 优先帮助尚未通过核心验收的学生。
- 检查扩展是否破坏输入验证、资源结算和任务日更新。
- 不直接输入整段答案；要求学生先指出期望路径。
- 记录可展示的不同设计，供主讲邀请自愿分享。

## 11. 3 分钟展示结构

1. 30 秒：基地名称、任务目标和自己的调度策略。
2. 60 秒：运行程序，展示至少两次决策和资源变化。
3. 60 秒：解释一个条件或循环如何控制流程。
4. 30 秒：说明一个错误、测试输入和修复方式。

展示者不逐行读代码，不播放无法复现的长流程。主讲可追问：

- 这个变量在什么时候变化？
- 输入无效值为什么不会消耗一天？
- 你的新规则用哪组输入验证？
- `break` 或 `continue` 在这里跳到哪里？

## 12. 评价量表

| 维度 | 比例 | 判断依据 |
| --- | ---: | --- |
| 可运行性 | 20% | 能启动、交互并正常结束 |
| 知识覆盖 | 25% | 正确使用输入、变量、条件和循环 |
| 逻辑正确性 | 20% | 常见输入和边界结果正确 |
| 调试与解释 | 15% | 有真实修复记录并能说明 |
| 可读性 | 10% | 命名、缩进、注释和输出清楚 |
| 创意与展示 | 10% | 改进有规则、有测试、能表达 |

核心行为错误不能用界面美化或故事创意抵消。功能少但完全理解、测试充分的作品应得到合理高分。

## 13. 同伴反馈与收尾

每位观察者只写两句话：

1. 我看懂了他的哪一个程序设计。
2. 我想追问哪一组输入或边界。

课后教师记录完成率、四类结果覆盖率、最常见逻辑错误、扩展功能成功率和实际展示人数，作为下一轮课程细节调整依据。
